## Monorepo Structure

This is a **Turborepo + npm workspaces** monorepo with three packages:

- `apps/frontend` — TanStack Start (React 19 + TypeScript) web app, served at port 3000
- `apps/studio` — Sanity Studio v5, served at port 3333
- `packages/shared` — Shared TypeScript types generated from Sanity schemas

## Commands

### Development
```bash
npm run dev                                    # Start all apps (frontend + studio + shared watch)
npm run dev --workspace=@santan/frontend       # Frontend only
npm run dev --workspace=@santan/studio         # Studio only
```

### Building
```bash
npm run build                                  # Build all packages
npm run build --workspace=@santan/frontend     # Frontend only
```

### Linting, Formatting, Type Checking
```bash
npm run lint          # Biome lint (all packages)
npm run lint:fix      # Biome lint with auto-fix
npm run format        # Biome format (all packages)
npm run check         # Biome check (lint + format)
npm run check:fix     # Biome check with auto-fix
npm run type-check    # TypeScript type check (all packages)
```

### Testing (Frontend)
```bash
cd apps/frontend && npm test                           # Run all tests
cd apps/frontend && npx vitest run src/path/to/test.ts # Run a single test file
```

### Sanity Types
After modifying Sanity schemas, regenerate TypeScript types:
```bash
cd apps/studio && npm run generate-types
```
This extracts the schema, generates types in `packages/shared/src/types/`, and must be followed by rebuilding shared: `npm run build --workspace=@santan/shared`.

## Architecture

### Data Flow

Content lives in **Sanity**. The frontend fetches it via **GROQ queries** defined in `apps/frontend/src/sanity/serverFunctions.ts` using TanStack Start's `createServerFn`. Loaders in `apps/frontend/src/loaders/` call these server functions via TanStack Query (`ensureQueryData`), then pass the data to route components.

```
Sanity CMS → GROQ query (serverFunctions.ts) → Loader (loaders/*.ts) → Route component (routes/*.tsx) → Page component (pages/*)
```

### Routing

File-based routing under `apps/frontend/src/routes/`. Each route file exports a `Route` via `createFileRoute` with a `loader`, `head` (for SEO meta), and `component`. The route tree is auto-generated at `src/routeTree.gen.ts` — do not edit this file manually.

Dynamic content (posts and categories) is handled by the catch-all route `routes/$.tsx`, which uses the `fullSlug` field to look up documents in Sanity regardless of nesting depth.

### FullSlug System

Posts and categories use a computed `fullSlug` field (e.g. `technology/web/my-post`) that encodes the full parent hierarchy. This is auto-computed on publish by `apps/studio/src/actions/fullSlugPublishAction.tsx` and cascades to all child documents. See `docs/FULLSLUG_SYSTEM.md` for details.

### Preview Mode

Loaders check for preview mode via `detectPreviewMode()`. In preview mode, the Sanity client switches to `perspective: 'drafts'` and uses a server-side `SANITY_READ_TOKEN`. The `@sanity/react-loader` with `useLiveMode` powers real-time visual editing overlays (Stega).

### Styling

Two systems are used together:
- **Vanilla Extract** (`.css.ts` files) — typed, scoped styles for components
- **Tailwind CSS v4** — utility classes

`.css.ts` files are listed as `sideEffects` in `apps/frontend/package.json` to ensure they are included in the bundle.

### Type Safety

Sanity query results are validated at runtime with **Zod** schemas defined in `apps/frontend/src/types/`. The `@santan/shared` package exports generated type literals (`sanityTypeLiterals`) used to discriminate document types from Sanity responses.

### Sanity Studio

Schema types are organized in `apps/studio/src/schemaTypes/`:
- `fieldTypes/` — reusable field definitions
- `blockTypes/` — portable text block types (accordion, image carousel, etc.)
- `index.ts` — registers all schemas
